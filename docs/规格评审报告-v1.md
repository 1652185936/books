## 1. 总体评价

> **骨架优秀、关节缺失。** 分层架构、安全意识、数据库表结构、状态机框架都达到专业水准；但「数据如何从一张表流到另一张表、业务流程每一步由谁触发、关键算法怎么算」几乎全部停留在"点了名却没定义"。直接据此实现，会在三类问题上集中爆雷：**功能跑不起来 / 多人实现打架 / 升级丢数据**。

| 维度 | 评级 | 主要病灶 |
|---|---|---|
| D1 数据流完整性 | 🔴 不合格 | 表间「搬运工」未定义：写入/创建路径缺失、协议↔DB 映射断裂、派生数据无生成规范 |
| D2 业务流完整性 | 🔴 不合格 | 状态机只画正常路径，缺取消/重试/升级/后台/恢复分支 |
| D3 描述清晰度 | 🟠 偏弱 | 大量「由导入服务保证」式甩锅，关键算法只有原则没有公式 |
| D4 设计规范性 | 🟡 中等偏上 | DB 设计扎实，但枚举缺 NOT NULL、唯一约束误伤、版本被 const 锁死、ReDoS 风险 |
| D5 接口/契约完整性 | 🔴 不合格 | 接口是草图、DTO 只有类型名无结构；F08/F10/F11/F12 几乎无接口 |
| D6 规格文档完整性 | 🟠 偏弱 | NFR 不可追踪、测试不可执行、缺术语表/i18n/统计公式/识别可行性论证 |

## 2. 五条根因（建议据此成批修，而非逐条打补丁）

1. **协议 ↔ DB 之间缺一张《字段映射 + 派生规范》表**：补它能一次性解决 D1 大半、D3 多条，以及 `contentVersion` 缺失。
2. **「创建/写入」路径系统性缺失**：`review_cards` 谁建、`wrong_items` 怎么从判分汇总、`learning_events` 谁写、笔记/高亮接口在哪——把「读」与「写」对称补齐。
3. **状态机只画了正常路径**：补全取消、重试、升级、后台继续、崩溃恢复的转移边与吸收态清理。
4. **算法层全是「原则」没有「公式」**：完成度、连续天数、有效时长、判分阈值、多空聚合、统计口径——每条给可执行定义（含时区/分母/边界）。
5. **约束与契约层有洞**：枚举统一 `NOT NULL`、唯一约束改为「标准化后去重而非报错」、版本号 `const` 改 `pattern`、补齐 DTO 结构与缺失 Schema。

## 5. D1-数据流完整性（12 条）

> 核心病灶不是「少了哪张表」，而是表与表之间的「搬运工」没定义：写入/创建路径缺失、协议→DB 映射断裂、派生数据无生成规范、升级时冗余列漂移。

#### [R-001] 🔴 阻断 ｜ review_cards 缺少『谁写入』的产生时机定义，复习卡从不被创建

- **位置**：03-LLD §3.19 review_cards / §4.2 章节学习流程图 "生成或更新复习卡" / §7 Repository（无 ReviewRepository.create）
- **问题**：所有文档只定义了复习卡的『读（getDueCards）』和『评分更新（applyRating）』，但从未定义复习卡是『何时、由谁、根据什么规则创建』。source_type 的 package/wrong-item/user 三种来源都没有对应的生成流程与接口，初始 due_at/ease_factor/state 的赋值时机也空白。
- **影响**：review_cards 是只读不写的孤儿表：getDueCards 永远查不到卡片，F09『今日到期卡片数量』恒为 0，整个间隔复习（§5 调度规则）无输入。错题→复习的 source_type='wrong-item' 转化路径也没有触发点。
- **建议修复**：明确定义复习卡生成规则与时机：何时由 wrong_item 生成卡（如错误后/达到阈值）、何时由 package flashcard 生成、是否允许用户手动建卡；补 ReviewRepository.createCard 接口和初始 due_at/ease_factor/state 取值，并在 §4.2/§4.5 标注创建动作。

#### [R-002] 🟠 重大 ｜ 协议 flashcards[] 无落库去向，复习卡来源 source_type='package' 成为悬空引用

- **位置**：chapter.schema.json L98 flashcards / AI-PDF学习包规范-v1.md §5 "flashcards": [] / 03-LLD §3.19 review_cards.source_type、§3.1 ER 图
- **问题**：协议在章节里定义了 flashcards 数组（包内自带复习卡），但数据库 §3 没有任何 flashcards 表，导入流程（§2.3 状态机 indexing 阶段、§7.1 BooksRepository.upsertImportedBook）也未列出写入 flashcard 的步骤。review_cards.source_type 允许 'package'，但 source_id 指向的包内 flashcard 内容无处存储，front_json/back_json 也无来源。
- **影响**：导入时章节 JSON 里的 flashcards 被静默丢弃（违反 OCR §10『不得静默丢弃内容』精神）；F09 复习卡片功能对『package 来源卡片』无数据可读，source_type='package' 永远造不出来，复习闭环的一个主要入口断裂。
- **建议修复**：二选一并写清：①新增 flashcards 表（或在 review_cards 直接落 front/back）并在导入 indexing 阶段写入 flashcard，明确 source_id=包内 flashcard external_id；②若第一版不支持包内 flashcard，则从 chapter.schema.json 和规范 §5 删除 flashcards，并从 review_cards.source_type 去掉 'package'。
- **核验校正**：需修正两处，故判 partially-correct 而非 confirmed：

(1) 严重度夸大。原文证据不支持 blocker：flashcards 不在 chapter.schema.json 的 required 列表（L6-16）内，是可选字段，规范示例本身即为空 `[]`；F09 复习闭环仍可由 wrong-item（练习判分流程 HLD §4.2 L158-160 自动生成）与 user 两个来源支撑，并非整体失效，仅 'package' 这一条来源路径成为死代码。应降为 major（一个枚举分支永远造不出 + 可选内容静默丢弃），而非 blocker。

(2) “front_json/back_json 无来源 / source_id 指向的内容无处存储”表述不准确。review_cards 自身就含 front_json/back_json 两列（§3.19 L549-550），若导入流程将 chapter.flashcards[] 转换为 review_cards 行，内容是有处存储的。真正缺的是“把 flashcards[] 转成 review_cards 行（source_type='package'）”这一 ETL 步骤，而非存储字段缺失。定位与方向正确，但“无处存储”的措辞应改为“缺少导入转换步骤”。

#### [R-003] 🟠 重大 ｜ 协议 partialRules（whenTokensEqual）与 DB grading_rules.rule_type 取值不对齐，判分规则落库无映射

- **位置**：AI-PDF学习包规范-v1.md §6 grading.partialRules[].whenTokensEqual / 03-LLD §3.11 grading_rules.rule_type、params_json
- **问题**：协议侧规则用 whenTokensEqual 描述（按 token 列表匹配），DB 侧 rule_type 是另一套枚举且新增了协议里不存在的必填 feedback_code。导入服务如何把 partialRules[].whenTokensEqual 映射到 (rule_type, params_json) 没有定义；feedback_code 在协议中无对应字段（协议只有 feedbackZh），却是 NOT NULL。
- **影响**：导入器写 grading_rules 时面对无法确定的映射：whenTokensEqual 该转 'tokens-equal' 还是 'missing-token'？feedback_code 无来源会导致 NOT NULL 写入失败或被迫塞空串/编造值。判分引擎（§4.4 第4步『按 priority 执行』）读取的规则语义与协议不一致，F07『缺少助动词 is』反馈可能丢失或判分结果错乱。
- **建议修复**：补一张协议 partialRules → grading_rules 的字段映射表：whenTokensEqual→rule_type+params_json 的确定规则、result→result、feedbackZh→feedback_zh；明确 feedback_code 的生成方式（如由 partialRule.id 派生或允许为空），priority 的来源（数组下标？）。

#### [R-004] 🟠 重大 ｜ 非手写题型（choice/true-false/match/order-words）缺少作答与判分数据落库路径

- **位置**：03-LLD §3.8 exercises 允许题型 / §3.9 exercise_blanks / §3.14 exercise_attempts / §3.15 handwriting_records
- **问题**：四种非手写题型没有 blank/笔迹/识别概念。它们的选项、正确项、用户选择如何存储未定义：exercises.prompt_json/context_json 是否存选项、accepted_answers 能否表达『选 B』『正确配对』、attempts 用哪个字段记录非识别型作答结果，全部空白。
- **影响**：实现 choice/true-false/match/order-words 时无表可落：用户选择无处存、判分无标准答案结构、attempts 的 recognition_status 对这些题无意义（强制 not-requested?）。这些题型要么无法实现，要么各自发明私有 JSON，破坏统一判分/错题/统计链路。
- **建议修复**：明确：①这些题型的选项与正确答案存哪（建议 prompt_json/context_json + accepted_answers 复用，并写出结构）；②attempts 增加一个与识别无关的 user_response_json 字段或说明复用 normalized_text_json 的约定；③说明 recognition_status 对非识别题取值。或在第一版范围内明确仅实现手写题、其余题型延后并从枚举移除。

#### [R-005] 🟠 重大 ｜ content_blocks.plain_text 与 source_pages_json 是派生数据，但生成时机、来源与 NOT NULL 约束无定义

- **位置**：03-LLD §3.6 content_blocks.plain_text、source_pages_json / AI-PDF学习包规范-v1.md §5.2 blocks
- **问题**：plain_text 需要从异构 block payload 派生（paragraph 取 text、table 拼 rows、example 拼 english+chinese…），但 LLD 未定义派生算法与时机（导入时？渲染时？）。source_pages_json 是 NOT NULL 但协议在 block 粒度没有来源页，只能从 section.sourceRef.pdfPages 下放——这个继承规则未写明。
- **影响**：不同开发者会用不同方式生成 plain_text，导致 F04/F11 搜索（FR-F04-04、ContentRepository.search 基于 plain_text 索引）结果不一致、table/formula 块搜不到；source_pages_json NOT NULL 在 block 无来源时只能塞空数组或继承 section，行为不统一，FR-F05-07『来源页内部追溯』失真。
- **建议修复**：在 LLD 增加『派生数据生成规范』：逐 block.type 定义 plain_text 提取规则、在导入 indexing 阶段一次性计算落库；明确 block.source_pages_json 继承自所在 section.sourceRef.pdfPages（或在协议中给 block 增加 sourcePages）。
- **核验校正**：建议修正影响表述：①不要把 plain_text 搜索不一致直接挂到 FR-F04-04（该需求是章节标题/中文标题/关键词检索，范围更窄）。②补充一个原文确凿的连带影响——§3.17 highlights 约束『range_end 不得超过保存时 plain_text 长度』（第525行）依赖 plain_text，派生口径不一致会使高亮锚点越界，这比搜索更能支撑 major 严重度。整体维持 major：派生算法/时机与 block 级 source_pages_json 继承规则缺失，确会导致多开发者实现分歧与数据不一致。

#### [R-006] 🟠 重大 ｜ exercise_attempts 冗余 book_id/chapter_id 在学习包升级（external_id 重映射）时的同步未定义

- **位置**：03-LLD §3.14 exercise_attempts.book_id/chapter_id『冗余便于查询』/ §9 数据升级 external_id 映射
- **问题**：升级覆盖内容时（FR-F03-06 覆盖保留学习记录），chapters/exercises 会按 external_id 重建为新内部 id 还是原地更新没说清。若新内部 id，则 exercise_attempts.exercise_id/chapter_id/book_id 这三个冗余 FK 如何重指向新行、保持一致，无维护规则。wrong_items.exercise_id、review_cards.chapter_id 同理。
- **影响**：升级后历史 attempts 可能指向被删除的旧 chapter/exercise 行触发 ON DELETE CASCADE，直接丢失全部答题历史与错题（违反 FR-F03-06『保留学习记录』、F10『历史不覆盖』、总体验收 5『重启后错题不丢失』）；或冗余 book_id/chapter_id 与新 exercise 的真实归属不一致，导致 F10/F11 按 book/chapter 统计串数据。
- **建议修复**：定义升级映射事务：以 external_id 为锚做 UPDATE 而非 delete+insert 保持内部 id 稳定；若必须换 id，给出 attempts/wrong_items/review_cards 外键与冗余列的批量重指向步骤，并对无法映射的练习定义降级策略（保留历史但标记 orphaned）。
- **核验校正**：机制描述需轻微修正：CASCADE 语义下，attempts 行会随父行 chapter/exercise 一起被级联删除，而非"指向被删除的旧行"（外键约束不允许悬空指向）。但结果判断正确——升级若走重建路径即丢失全部答题历史/错题，违反 FR-F03-06 与验收5。建议设计文档明确二选一：(a) 升级用原地 UPDATE 保持内部 id 不变；或 (b) 若必须重建，须给出 external_id→新内部 id 的重映射步骤，在删旧行前先把 exercise_attempts.exercise_id/chapter_id/book_id、wrong_items.exercise_id、review_cards.chapter_id 改指新行（或临时解除 CASCADE/用事务迁移）。严重度 major 维持不变。

#### [R-007] 🟠 重大 ｜ wrong_items.error_type / correct_streak 的产生与更新规则未定义，错题闭环数据来源缺失

- **位置**：03-LLD §3.21 wrong_items.error_type、correct_streak、status / §4.5 判分结果『加入错题』 / 01-SRS FR-F10-06
- **问题**：error_type 的取值集合（枚举？自由文本？）与产生它的数据来源没定义——判分只产出 grade(correct/partial/incorrect) 和 feedback_code，没有 error_type 字段，二者如何对应未说明。correct_streak 在何处递增/清零、mastered 阈值（FR-F10-06『配置阈值』）配在哪（user_settings 无此键）也缺失。
- **影响**：开发者无法确定『加入错题』时 error_type 填什么、是否枚举，导致 F10 按错误类型筛选（FR-F10-02）数据不可枚举；correct_streak/mastered 无更新规则，FR-F10-06 无法落地；阈值无配置项导致掌握判定硬编码或缺失。
- **建议修复**：定义 error_type 受控词表并给出『grade/feedback_code → error_type』映射；写明错题 upsert 规则（首次创建、重复 error_count+1、重做正确 correct_streak+1 否则清零、达阈值置 mastered）；在 user_settings 增加 wrongbook.masteryStreak 键（FR-F10-06 的『配置阈值』）。

#### [R-008] 🟡 一般 ｜ report.json（reviewQueue/warnings/quality.issues）无任何落库表，IMP-015 警告与人工复核内容导入后丢失

- **位置**：AI-PDF学习包规范-v1.md §8 report.json reviewQueue/warnings、§5 quality.issues / 03-LLD §2.5 IMP-015、§3 全部表
- **问题**：report.json 被复制进正式目录却没有索引到数据库，import_jobs.warning_count 只存了个计数。reviewQueue 的逐条复核项（哪个 jsonPath 的内容不可信）导入后无结构化去向。

#### [R-012] 🟡 一般 ｜ FR-F05-05『收藏当前章节』与 bookmarks 表结构（依赖 block_id）矛盾，章节级收藏去重失效

## 6. D2-业务流完整性（12 条）

> 状态机画了框，但缺了一半的边和恢复路径：取消、失败重试、升级、后台继续、崩溃恢复、多空题聚合、识别超时退出。

#### [R-013] 🔴 阻断 ｜ 导入状态机与校验顺序对解压时机的定义自相矛盾

- **位置**：03-详细设计说明书.md §2.3 导入状态机 / §2.4 校验顺序
- **问题**：状态机把 extracting 放在 awaiting_confirmation（用户确认）之后，但 §2.4 把解压(step6)放在 SHA-256(step7)/Schema(step8)/业务(step9)之前。这意味着 validating_hash/validating_schema/validating_business 三个状态在状态机里发生于 extracting 之前，而它们实际依赖已解压的文件内容。两份文档对『何时解压』定义直接冲突。
- **影响**：开发者无法确定解压时机：若按状态机在确认后才 extracting，则 hash/schema/business 阶段无文件可校验，等于跳过 BR-002 安全校验；若按 §2.4 在校验前解压，则状态机 extracting 状态语义重复且确认前已落盘，违背 FR-F03-04『校验通过后由用户确认才写入』的隔离假设。
- **建议修复**：统一时序：在 manifest/版本校验后先解压到隔离暂存目录，再做 hash/schema/business 校验；awaiting_confirmation 后的步骤改为『原子提升到正式目录/索引』而非二次解压。在状态机中显式区分 extracting_to_staging（确认前）与 promoting_to_final（确认后），并使 §2.4 步骤与状态名一一对应。

#### [R-014] 🟠 重大 ｜ cancelled 仅能从 selected 转出，无法表达校验中/确认前取消，与 FR-F02-05 矛盾

- **位置**：03-详细设计说明书.md §2.3；对照 01-需求规格说明书.md FR-F02-05、02-概要设计说明书.md §6 导航约束
- **问题**：需求要求在『写入数据库前』的任意阶段（copying/validating_*/awaiting_confirmation/extracting）都能取消，但状态机只给了 selected→cancelled 一条取消边。validating 阶段与 awaiting_confirmation 没有到 cancelled 的转移，cancel(jobId) 调用将落到一个没有合法目标态的状态。
- **影响**：用户在校验中或确认前点取消时状态机无合法转移，实现要么非法改状态、要么被迫走 failed（语义错误，会被当成错误上报或计入 warning_count），且取消时 temp/import/<jobId> 与半成品 import_jobs 记录的清理路径未定义，可能残留垃圾文件与脏记录。
- **建议修复**：为 copying、validating_zip…validating_business、awaiting_confirmation、extracting 均补 `--> cancelled` 边，并定义 cancelled 的统一清理动作（删暂存目录、回滚未提交事务、保留脱敏摘要）。明确 indexing/committing 之后不可取消。
- **核验校正**：核心问题（cancelled 仅从 selected 转出、与 FR-F02-05「写入数据库前取消」矛盾、cancel(jobId) 在校验中/确认前无合法目标态、中途取消清理路径未定义）全部经原文证实，方向与定位准确，严重度 major 合理。唯一需修正的细节：原文证据中「会被计入 warning_count」表述有误——warning_count 字段（§3.22）服务于 IMP-014/015 这类非阻断警告，失败上报实际写入的是 error_code/error_detail，与 warning_count 无关；这一处夸大/定位偏差应剔除，但不影响发现整体成立。建议落地修复：在 §2.3 状态机为 copying、validating_*、awaiting_confirmation 增加 `--> cancelled` 边，并补充与「失败处理」并列的「取消处理」清理流程（删 temp/import/<jobId>、回滚未提交事务、清理 package_files 与 import_jobs 半成品行、置 state=cancelled 而非 failed）。

#### [R-015] 🟠 重大 ｜ 失败后无法重试：状态机无 failed→重试 的转移，与 FR-F07-09 重试语义混淆且 P-F03 仅说『重试创建新阶段记录』

- **问题**：IMP-013（存储空间不足）和 IMP-016（数据库提交失败）属于可恢复失败，用户清理空间或重试应能继续，但状态机把它们和不可恢复失败（路径穿越、哈希不符）一样落入吸收态 failed，没有任何重试转移。P-F03『重试创建新阶段记录』究竟是新建 import_job 还是复用，未定义；source_uri 只存哈希（§3.22 source_uri_hash），重试时能否重新拿到原始 URI 也未说明。
- **影响**：存储不足或瞬时 DB 失败后用户只能从头重新走系统文件选择器（且原 URI 已不可用，DocumentViewPicker 授权可能失效），体验断裂；开发者不知道重试是复用 jobId 还是新建，可能导致 import_jobs/package_files 主键冲突或孤儿记录。
- **建议修复**：区分可重试失败与终态拒绝：为 IMP-013/IMP-016 等定义 failed→retrying→对应阶段 的转移或显式『新建 job 复制旧 source 引用』策略；明确重试时 URI 持久授权（如保存可重开的持久化 URI 而非仅哈希）。

#### [R-016] 🟠 重大 ｜ 升级(upgrade)流程缺少独立状态机与回滚点产生时机，books.status=updating 无对应导入态

#### [R-017] 🟠 重大 ｜ HLD『导入离开页面后台继续』在状态机与持久化层无落地：UI 关闭后由谁驱动 validate/commit 未定义

#### [R-018] 🟠 重大 ｜ 多空题部分识别成功/部分失败时 attempt.grade 与 recognition_status 的聚合规则缺失

- **位置**：03-详细设计说明书.md §4.2 识别状态机、§3.14 exercise_attempts、§4.5 判分结果；对照 01 FR-F06-04、FR-F07-09
- **问题**：multi-blank-handwriting 一道题有多个 blank，可能 blank-1 识别成功、blank-2 识别失败。但 attempt 只有单个 recognition_status 与单个 grade，文档没有定义：整题 recognition_status 如何由各空聚合？某空 recognition-failed 时整题 grade 是判 recognition-failed（阻止其他空判分，违背已成功空的反馈）还是 partial？§4.5 也只给单值结果映射。

#### [R-019] 🟠 重大 ｜ 非手写题型(choice/true-false/match/order-words)的作答与判分全链路缺失

- **位置**：AI-PDF学习包规范-v1.md §6.1、chapter.schema.json exercises.type、03 §3.8 题型、§3.15 handwriting_records、§4 手写与识别设计

## 7. D3-描述清晰度与歧义（13 条）

> 把复杂逻辑甩给一句「由导入服务保证」，关键算法只有原则没有可执行公式——会导致多人实现口径不一致、验收无法判定。

#### [R-025] 🔴 阻断 ｜ completion_percent 完成度算法完全缺失，无可执行定义

- **位置**：03-详细设计 §3.12 reading_progress.completion_percent；01-需求 FR-F04-01；FR-F11-01 掌握章节数
- **问题**：completion_percent 是「按已读小节/总小节」「按已读内容块」「按已做练习」还是「按阅读滚动位置」完全没定义，也没说何时由谁写入、增量还是覆盖。书籍级完成度与章节级完成度的聚合关系也没写。
- **影响**：不同开发者会用不同口径计算，导致书架进度条、目录完成度、统计页「掌握章节数」三处数字互相打架；重算（FR-F11-05 要求可重算）时无法对齐，验收『重启后进度一致』无法判定通过。
- **建议修复**：明确 completion_percent 的分母（小节数/内容块数/练习数）、累计触发事件（阅读到底部？停留？做题？）、写入时机与四舍五入规则；并定义书籍级完成度由章节完成度如何聚合。

#### [R-026] 🟠 重大 ｜ 章节状态 not-started/learning/completed/review-due 的判定条件未定义

- **位置**：03-详细设计 §3.12 reading_progress.status；01-需求 FR-F04-03
- **问题**：「已完成(completed)」是读完讲解即算，还是必须做完练习且全对？「需复习(review-due)」是由 review_cards.due_at 到期驱动、还是 wrong_items.next_due_at 驱动、还是独立计算？状态机迁移规则缺失。
- **影响**：目录页章节标识、统计页『掌握章节数』、复习入口三处对『完成』口径不一致，会出现『讲解读完显示已完成但练习全错』或『review-due 永不触发』等 BUG。
- **建议修复**：给出每个状态的进入条件（如 completed = 该章全部 enabled 练习最近一次 grade=correct 且讲解全部小节已读），并定义 review-due 由哪个表的到期字段驱动。
- **核验校正**：建议将自评严重度由 blocker 降为 major。该问题是真实且影响目录页标识、统计「掌握章节数」、复习入口三处口径一致性的设计缺口，但属于可派生/可重算的展示态（非数据丢失：FR-F11-05 明确统计可从原始 learning_events 重算，错题/复习另有 wrong_items.next_due_at 与 review_cards.due_at 作为真实驱动源），不会阻断核心学习闭环，故不构成 blocker 级别。修正点仅在严重度，问题描述与定位本身成立。

#### [R-027] 🟠 重大 ｜ 连续天数(streak)与有效学习时长算法欠定义，时区/边界未定

- **位置**：01-需求 FR-F11-01/FR-F11-04；03-详细设计 §3.13 learning_events、§11.1
- **问题**：(1) 连续天数按哪条时区的『自然日』切分未写——时间全存 UTC，但『连续天数』对用户是本地日历概念，跨零点会算错。(2)『连续天数』的判定门槛（当天需学习多久/做几题才算有效一天）未定义。(3) 有效时长：进入暂停前那 2 分钟无操作是否计入、心跳事件如何拼接成『时长』也没给公式，仅说不累计。
- **影响**：streak 在 UTC 与本地时区边界会出现+1/-1 偏差；不同开发者对『有效一天』门槛取值不同，统计页连续天数和验收『误差不超过 1 分钟』无法稳定通过。
- **建议修复**：明确按设备本地时区切日并记录时区；给出『有效一天』的最小门槛；给出由 learning_events 聚合有效时长的精确规则（暂停前空窗是否计入、心跳合并窗口）。

#### [R-028] 🟠 重大 ｜ 判分『判错 vs 仅提示』缺少可执行编辑距离阈值，与协议 maxEditDistance 冲突

- **位置**：03-详细设计 §4.4 判分优先级；协议 §6 grading.maxEditDistance；01-需求 FR-F07-07
- **问题**：maxEditDistance 在协议里是数字阈值但语义只说『不能判正确』——那么编辑距离≤阈值时究竟判 incorrect 还是 partial？是仅加一句提示文案、还是改变 grade 结果？『内置语法规则(§4.4 第5步)』具体有哪些、优先级如何与 partialRules 交错也没列出。
- **影响**：验收用例『is thinking 不得因编辑距离接近判正确』可过，但『drinkin(g漏笔)』这类该判 partial 还是 incorrect 全凭实现者猜，判分结果不可复现，错题本归类随之漂移。
- **建议修复**：明确 maxEditDistance 命中时的确切 grade（建议 incorrect + 提示，不改 score），并把『内置语法规则』列成可枚举的规则清单与优先级。
- **核验校正**：substance 成立、严重度 major 合理、定位准确，建议维持为有效发现，但需修正标题与描述中的"冲突"措辞：协议 §6.2 与 LLD §4.4 实为一致（都说编辑距离只产生提示、不得判 correct），并不矛盾；真正问题是二者共同欠定义——(1) 编辑距离≤阈值且非精确匹配时 grade 究竟落 incorrect 还是 partial 未定；(2) 该"提示"是否改变 grade 还是仅叠加文案未定；(3) §4.4 第5步"内置语法规则"具体规则集及其与学习包 partialRules 的优先级交错未列举。建议改写为"判分『incorrect vs partial』缺少可执行阈值语义，且内置语法规则未定义"。

#### [R-029] 🟠 重大 ｜ 答案标准化与判分耦合的开关来源不清：题级 grading 字段未映射到 DB

- **位置**：协议 §6 grading(caseSensitive/ignoreTerminalPunctuation/normalizeApostrophes 等)；03-详细设计 §4.3、§3.8 exercises、§3.11 grading_rules
- **问题**：协议的 grading 标量开关（caseSensitive 等）落到哪张表哪个字段没写——是塞进 exercises.feedback_json/context_json，还是 grading_rules.params_json，还是丢失？§4.3 标准化步骤依赖这些开关，但数据通道断了。
- **影响**：导入服务不知道把 caseSensitive 等写到哪里，判分引擎读不到配置只能用硬编码默认，导致『题目要求大小写敏感』被忽略，判分错误且无法追溯。
- **建议修复**：在 exercises 表新增 grading_options_json 或明确这些开关进入哪一列，并在 §4.3/§4.4 注明判分引擎从该列读取标准化开关。

## 8. D4-设计规范性（12 条）

> 数据库设计整体扎实，但有几个硬洞：枚举缺 NOT NULL 致 CHECK 被 NULL 穿透、唯一约束误伤合法数据、版本被 const 锁死、不可信正则 ReDoS、软删除策略割裂、缺关键 Schema。

#### [R-038] 🟠 重大 ｜ 软删除策略不一致：books/notes/highlights 有 deleted_at，但其它用户数据表全部缺失，导致删除语义割裂

- **位置**：03-详细设计说明书.md §1.3、§3.3 books、§3.16 notes、§3.17 highlights、§3.18 bookmarks、§3.14 exercise_attempts、§3.21 wrong_items、§3.19 review_cards
- **问题**：books 同时存在软删除（status=deleted / deleted_at）与硬删除外键级联（chapters/exercises/attempts 等 ON DELETE CASCADE books.id）两套语义，且没有定义谁触发谁。若按 status='deleted' 软删 books，则下游 attempts/wrong_items/notes 不会被清理；若真正 DELETE books 行，则 ON DELETE CASCADE 会连同 attempts、wrong_items、review_cards 物理删除——而这些表没有 deleted_at 无法软删。删除一本书是否保留错题/笔迹（FR-F10『删除书籍时对应错题按用户选择删除或随备份导出』、BR-010）完全无法在当前 schema 下实现。
- **影响**：实现期会出现：软删的书在书架隐藏但其 attempts/统计事件仍参与统计聚合导致数据虚高；或硬删书时级联清空用户练习历史违反 FR-F10-02『重做后历史错误次数不被覆盖』和 BR-010 的『可选保留』。delete 行为不可预测，易丢用户数据。
- **建议修复**：明确二选一：①books 走纯软删除（status=deleted/deleted_at），所有下游查询统一加 book.deleted_at IS NULL 过滤，物理清理交由后台 GC，并把级联改为应用层处理；或②定义『删除书=物理 CASCADE』并给 exercise_attempts/wrong_items/review_cards 增加导出钩子。同时为所有用户可保留数据表补齐 deleted_at 或明确声明『本表不软删，随书 CASCADE』。
- **核验校正**：设计缺陷真实成立、严重度 major 准确，定位（§1.3、§3.3、§3.13/3.14/3.16-3.21）全部核对无误。仅需修正引用细节：FR-F10-02 实为「按...筛选」而非「历史错误次数不被覆盖」（应为 FR-F10-06「保留历史记录」）；「可选保留/随备份导出」是 SRS 第276行的独立 bullet，并非 FR-F10 编号项。修正这些引用后该发现可直接采纳。

#### [R-039] 🟠 重大 ｜ 枚举字段大量缺失 NOT NULL，CHECK 允许 NULL 穿透，违反 §1.3 双重约束要求

- **位置**：03-详细设计说明书.md §3.4 chapters.kind/review_status、§3.5 sections.type、§3.8 exercises.answer_source、§3.9 exercise_blanks.handwriting_mode、§3.11 grading_rules.rule_type/result、§3.12 reading_progress.status、§3.14 exercise_attempts.recognition_status/grade、§3.15 handwriting_records.target_type、§3.16 notes.type/relocation_state、§3.19 review_cards.source_type/state、§3.21 wrong_items.status、§3.23 package_files.validation_state
- **问题**：SQLite/relationalStore 的 `CHECK(col IN (...))` 在 col 为 NULL 时结果是 NULL（不为 FALSE），因此 NULL 可以绕过 CHECK 写入。这些枚举列没有 NOT NULL，意味着 kind、status、grade、result、rule_type 等关键状态列可被写成 NULL，而 ArkTS 联合类型不含 null，读出时 switch/匹配会落空。
- **影响**：reading_progress.status=NULL 时章节状态无法渲染（F04 章节状态丢失）；exercise_attempts.grade=NULL 与 'draft' 语义混淆，错题/统计聚合 WHERE grade IN('incorrect','partial') 会漏掉 NULL 行；grading_rules.result=NULL 导致判分流程取不到结果。运行期出现大量空状态分支未覆盖的崩溃或静默错误。
- **建议修复**：为所有枚举列统一补 `NOT NULL`（确需可空的如已说明的 grading_rules.blank_id 除外），形成 `TEXT NOT NULL CHECK(x IN (...))`，与 books.status 写法对齐，并在生成迁移 SQL 时做模板化检查。

#### [R-040] 🟠 重大 ｜ accepted_answers 唯一约束基于 normalized_answer，标准化后多个可接受答案会冲突丢失

- **位置**：03-详细设计说明书.md §3.10 accepted_answers UNIQUE(blank_id, normalized_answer) + §4.3 标准化；AI-PDF学习包规范-v1.md §6 answers.accepted
- **问题**：示例里 `'s drinking`（直撇号）与 `’s drinking`（弯撇号）经 §4.3 第4步『弯引号统一为直引号』标准化后 normalized_answer 完全相同，写入第二条会违反 UNIQUE(blank_id, normalized_answer)。同理大小写不敏感时 `Is drinking`/`is drinking` 也会归一冲突。规范鼓励作者列多个变体，但导入层的唯一键恰恰把这些合法变体判为重复。
- **影响**：导入一个语法上完全合法、规范示例本身就给出的学习包时，第二、三条 accepted 答案要么写入失败导致整包导入回滚（BR-003），要么被静默去重导致 display_answer 丢失。规范自带示例无法通过自家导入逻辑，自相矛盾。
- **建议修复**：将唯一键改为 UNIQUE(blank_id, normalized_answer, display_answer) 或在导入时对标准化后重复的 accepted 做去重合并并保留首个 display；或明确规定 normalized_answer 去重是预期行为且 display 取首个。需在规范 §6.2 与 LLD §3.10 之间对齐。

#### [R-041] 🟠 重大 ｜ handwriting_records 同时挂 exercise_attempts 和 notes，两条 CASCADE 外键＋互斥约束无法用纯声明式 FK 表达

- **位置**：03-详细设计说明书.md §3.15 handwriting_records

#### [R-042] 🟠 重大 ｜ manifest/book/chapter Schema 用 const 锁死版本，与 §8.5『语义化版本＋主版本白名单』『忽略未知可选字段』的兼容策略直接冲突

#### [R-044] 🟠 重大 ｜ regex 与 custom-code 判分规则来自不可信包，缺少复杂度/超时/沙箱执行约束，存在 ReDoS 风险

#### [R-045] 🟡 一般 ｜ content_blocks.plain_text 建普通索引用于搜索，但 ContentRepository.search 是关键词模糊查询，B-Tree 索引对 LIKE '%kw%' 无效

#### [R-046] 🟡 一般 ｜ 命名风格在 DB(snake_case)、JSON(camelCase)、枚举值(kebab-case)、路由(混合)间不统一且无映射约定

- **位置**：03-详细设计 §3 各表字段 vs 规范 §3-§6 JSON 字段；LLD §6 路由；枚举值散布各处
- **问题**：三套命名风格本身是常见且可接受的（DB snake、JSON camel、枚举 kebab），但文档从未给出权威映射规则（如『DB snake_case ↔ JSON camelCase 自动转换』），导入映射层需要逐字段手写转换。更隐蔽的是枚举值跨层完全一致地使用 kebab-case 字符串（如 answer_source 存 `ai-inferred`），ArkTS 联合类型也得用带连字符的字符串字面量，易因大小写/连字符笔误产生 CHECK 失败。

- **建议修复**：在 LLD §1.3 增加一节『跨层命名映射规则』：DB snake_case、JSON camelCase 由统一 mapper 自动转换；枚举字面量集中定义为 ArkTS const 联合类型并由该常量同时生成 DB CHECK 列表，避免手写漂移。
- **核验校正**：需修正「路由(混合 kebab/camel/query 混用)」这一处的定位/措辞，存在出入：核对 LLD §6 全部 12 个页面路由——`bookshelf`、`import/select`、`import/validate/{jobId}`、`book/{bookId}`、`lesson/{bookId}/{chapterId}?section={sectionId}`、`review?bookId={optional}`、`statistics?bookId={optional}&range=week`、`settings/{section}` 等，路径段一律为小写或 camelCase 参数 + query string，并没有任何 kebab-case 路由段；这是 Web 路由的常规且内部一致的写法，不构成「kebab/camel 混用」的命名漂移问题。因此把路由纳入「命名风格不统一」证据是夸大的，应删除路由这一支，发现的实际范围应收缩为「DB(snake) / JSON(camel) / 枚举值(kebab) 三层无字段名映射约定 + 枚举字符串易笔误」。结论方向正确，但证据面被略微放大，故判 partially-correct；severity 维持 minor。

#### [R-047] 🟡 一般 ｜ books.chapter_count 与 chapters 记录数靠『导入服务校验』维护，存在反规范化冗余且无同步约束，升级后易失真

- **影响**：若目标设备无法提供达标的离线英文手写识别,F06/F07主流程与§10总体验收第3、4条直接落空,属于项目级落地风险。

#### [R-066] 🟠 重大 ｜ 国际化/多语言资源规范缺失,与CLAUDE.md『所有可见文本进资源文件』约束无法对账

- **位置**：CLAUDE.md §5；03-LLD §1.2代码模块树 / §6通用页面约束
- **问题**：声明的强制约束在详细设计里没有落地载体——无资源目录、无键命名约定、无i18n方案。
- **影响**：开发者无处放置字符串、无键命名约定,极易回退成硬编码,违反CLAUDE.md最高约束且无法在评审中对账。
- **建议修复**：在LLD新增i18n资源结构章节:resources目录与base/zh_CN/en_US分层、字符串键命名规范、错误码→文案映射放置位置、应用UI语言策略;明确学习包内内容文案(feedbackZh等)与应用UI文案的边界。

#### [R-067] 🟠 重大 ｜ 统计聚合公式与口径未定义,统计表设计不足以支撑F11全部指标

- **位置**：01-SRS §6.11；03-LLD §3.13 learning_events / §6.11 P-F11
- **问题**：『可重算』『误差≤1分钟』是验收项却无公式,且无结构化的语法点/知识点维度表。
- **影响**：不同开发者算出的连续天数/掌握度/正确率会不一致且无法判定对错;薄弱知识点缺少受控词维度,F11 §3薄弱知识点与点击跳转无法可靠实现。
- **建议修复**：补统计口径章节:定义每个指标的计算公式、时区与切日规则、正确率分子分母(partial/recognition-failed计法)、连续天数定义、掌握度判定阈值;为『语法点/知识点』引入受控词表或维度表,使薄弱知识点可聚合可跳转。

#### [R-068] 🟡 一般 ｜ SRS缺失术语表/词汇表,核心业务术语无定义

- **位置**：01-需求规格说明书.md(全文,无§术语表章节)
- **问题**：SRS没有术语表,核心业务名词在不同文档中含义靠读者自行脑补。
- **影响**：开发者对『掌握度』『有效学习时长』『目标设备』的理解会各不相同,导致统计口径、性能基线、判分语义实现各异;测试无法据此判定是否符合需求。
- **建议修复**：新增SRS §术语表,逐条定义:语法感知判分、部分正确、有效学习时长、掌握度/掌握章节、薄弱知识点、连续天数、目标设备(具体机型/SoC/分辨率)、测试集、待重新定位、回滚点等。
- **核验校正**：建议保留该发现但收敛证据与严重度。真正未定义、构成实现/验收风险的是：掌握度、薄弱知识点、连续天数（统计口径，含跨天/时区边界）、目标设备（性能基线机型）、约定英文单词和短句测试集（90% 准确率的判定依据）。应从证据中删除「有效学习时长」「部分正确」「语法感知判分」三项——它们已被 FR-F11-04 / FR-F07-04 + §6.7验收 / FR-F07-04~07 内联定义或锚定。因最具破坏性的举例（有效学习时长）实为已定义，且包驱动语义已锚定，残余风险范围比原文所述窄，严重度由 major 下调为 minor。

#### [R-069] 🟡 一般 ｜ V001初始迁移脚本范围与迁移列表自相矛盾,初始建表内容不明确

- **位置**：03-LLD §9数据升级(迁移命名示例);§3数据库设计(约23张表)
- **问题**：开发者无法确定首版迁移脚本应建哪些表:照搬V001/V002/V003拆分则与books.content_version『V003才加』冲突;若V001一次建全表则§9示例具误导性。
- **影响**：直接导致建表脚本写错或迁移顺序错乱,V001范围歧义会让初始DDL无法落地。
- **建议修复**：明确声明V001的建表范围(建议第一版全部23张表一次性建立),并标注§9的V002/V003仅为『未来迁移命名风格示例、非本版任务』,或反之给出每个V脚本的确切DDL范围。
- **核验校正**：建议判 partially-correct 并降级为 minor。真实问题应收窄为：§9 的命名示例与 §3 的「单版完整 schema」并存时，文档未一句话点明「这是命名格式示例，首版由 V001 一次建全表」，存在被字面误读为迁移分层计划的歧义。修正点：(1) 严重度由 major 降为 minor——§3 是权威 schema，不会真正导致 DDL 无法落地；(2) 「自相矛盾」措辞不准，应表述为「命名示例与首版 schema 的关系未交代清楚，易被误读」，而非两处定义互相冲突；(3) 「照搬 V001/V002/V003 拆分会与 books.content_version 冲突」是基于把示例当计划的最坏读法，属潜在误读风险而非文档明确要求的拆分。

#### [R-070] 🟡 一般 ｜ 测试设计只列『测什么』未给输入/期望输出与通过判定,不可执行

- **位置**：03-LLD §11测试设计(§11.1单元/§11.2集成/§11.3 UI验收)
- **问题**：测试章节不可执行:QA无法据此写出确定性测试,『通过/失败』靠主观判断。
- **影响**：与需求无traceID关联,等于没有验证设计;CLAUDE.md『有测试或明确自动化验证』的完成定义无法满足。
- **建议修复**：将每条测试细化为『用例ID/关联需求/输入/步骤/期望输出/通过判定』,并提供官方示例学习包(含115章、含各题型、含损坏/越权样例)作为集成测试基准夹具(samples/目前为空)。

#### [R-071] 🟡 一般 ｜ 缓存策略(LRU/容量/失效)与笔迹/备份文件版本升级缺设计

- **位置**：02-HLD §10；03-LLD §2.2 cache/ 目录 / §4.1 笔迹文件version:1 / §7.2 BackupService

- **位置**：03-LLD §2.1 ZIP资源限制表；01-SRS §8.1 NFR-P-005

## 11. 生产端评审：AI 解析 PDF → 生成 `.studybook.zip`（是否缺失 + 是否可行）

> 这是整个产品的**第一前提**：没有合规的 ZIP，App 的 12 个功能都是空壳。本节为独立技术评审，已**剔除版权/上架/运营**（自用应用，不涉及）。

### 11.1 缺失（生产链当前几乎空白）

| 缺失项 | 现状 | 后果 |
|---|---|---|
| 确定性转换器 | 不存在任何转换代码，整条链只靠协议规范 §12 一段交给 AI 的自然语言提示词 | 转换非确定性，同书两次产出可能不同 |
| 校验器 validator | `tools/studybook-validator/` 只有 README、**未实现** | 规范 §11「导入前强制验收」沦为「AI 自称我验过了」，无独立闸门 |
| 样例包 | `samples/` 为空，**全项目无一个真实 `.studybook.zip`** | 无回归基准，App 端导入/校验逻辑无法对真实数据测试 |
| 关键 Schema | 仅 manifest/book/chapter 三份；report/checksums/exercise/grading/answers/flashcard/stroke 全无 | AI 输出的练习/答案/复核报告无强结构约束 |
| 产销契约 | `contentVersion` 协议里根本没有，但 App 靠它做去重/升级/回滚 | AI 不知该写什么；App 升级闭环无输入 |
| 复核回写闭环 | `report.json.reviewQueue` 产生后由谁复核、如何回灌、怎样定稿再生成版本——无定义 | 「AI 产→人复核→修订→再导入」回路断裂 |
| 转换质量门 | 90% 是 App 侧识别率；转换准确率/答案正确率无可度量验收 | 无法判定一个包「转得够不够好」才放行 |

### 11.2 可行性（分两种生产环境）

**假设 A：纯 LLM（只输出文本）——🔴 不可行，有物理死结**

- ❌ 生成 `checksums.json` 真实 SHA-256：LLM 无法可靠计算哈希，必然编造 → 哈希校验(IMP-009)必假。
- ❌ 「交付一个 ZIP」：LLM 产文本，产不出真正的二进制压缩包。
- ❌ 图片资产 + 其二进制/哈希/尺寸：产不出可校验的二进制图片。
- ⚠️ 115 章全局 ID 唯一 / `toc↔files` / `chapterCount` / 连续编号一致：长文本下极易漂移。

**假设 B：LLM + 工具（Codex/带代码执行 + OCR + 打包脚本）——🟡 原理可行，但按现有文档仍落不了地**

| 环节 | 可行性 | 说明 |
|---|---|---|
| 算 SHA-256 / 打包 zip | ✅ | 交给脚本，绝不交给 LLM 本体 |
| 扫描版 OCR | 🟡 | 大小写/标点可保，但语法表格/数学公式/版面准确率现实有限 |
| 「禁止凭空补写」vs「禁止静默丢弃」 | ⚠️ | OCR 不确定时二者天然冲突，强依赖人工复核兜底 |
| 文生图统一画风(115 章) | 🟡 | 一致比例/线条/色板 + 可控尺寸难稳定；建议走「插图可暂缺 + alt 降级」路线（规范 §7 已允许只存语义/提示，App F05-06 有 alt 降级） |
| 从书后答案抽标准答案不幻觉 | 🟡 | 可行但有幻觉，靠 `answerSource` 标注 + `ai-inferred` 入复核 |
| 规模/分批合并 | 🟡 | 分批可行，但全局一致性必须由确定性 validator 校验——而 validator 未实现 |
| 可复现 / `content_version` 稳定 | ⚠️ | LLM 非确定性 → 同书两次产出结构/ID/答案不同 → App 升级/回滚(FR-F03-06、§9)失效。自用也照样踩 |

### 11.3 裁决

| 结论 | 评级 |
|---|---|
| 纯 LLM(A) | 🔴 不可行（checksums / zip / 二进制三道物理墙） |
| LLM+工具(B)，按**现有文档** | 🔴 仍不可行（缺 validator / Schema / contentVersion / 复核回写 / 质量门） |
| LLM+工具(B)，**补齐下述前提后** | 🟢 可行（OCR / 文生图 / 无幻觉 / 可复现仍是需管理的软风险） |

### 11.4 要让生产端成立，必须补的 7 件事

1. 明确生产环境 = **LLM + 工具**，并把 **checksums / zip 打包 / 哈希计算交给确定性脚本**，绝不让 LLM 本体产。
2. **实现 validator**（电脑端确定性校验器），作为导入前真正的 gate，取代「AI 自验」。
3. **补齐 Schema**：report / checksums / exercise / grading / answers / flashcard / stroke，并把 `chapter.schema.json` 里 `prompt/answers/grading` 的 `type:object` 展开成真实结构。
4. **协议补 `contentVersion`**，并把「转换→人工定稿→固定 content_version」做成可复现流程。
5. **图片走「可暂缺 + alt 降级」**，不强制 AI 文生图，绕开一致性陷阱。
6. **定义复核回写闭环**：reviewQueue → 谁复核 → 怎么回灌 → 再定稿。
7. **建转换质量验收集** + 提交 **1 个真实样例包**到 `samples/` 作回归基准。
